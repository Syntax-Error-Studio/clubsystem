# 校园社团活动报名与签到管理系统 — 答辩速成手册

> 本手册按"老师可能问什么 → 你怎么答"来组织。每一段都是你能直接背的话术。

---

## 一、一句话说清项目

"这是一个基于 **Spring MVC + MyBatis + JSP** 的校园社团活动报名与签到管理系统。用户分三种角色（学生、社团管理员、超级管理员），学生可以浏览活动、报名、签到、查看公告；管理员可以管理活动、审核报名、操作签到、发布公告。数据库用 MySQL，构建用 Maven，部署到 Tomcat。"

## 二、整体架构（必问！）

### 2.1 技术栈一句话版

| 层 | 技术 | 作用 |
|---|---|---|
| 表现层 | JSP + JSTL | 页面渲染 |
| 控制层 | Spring MVC（DispatcherServlet） | 接收请求、调用Service、跳转页面 |
| 业务层 | Spring @Service + @Transactional | 业务逻辑、事务管理 |
| 持久层 | MyBatis（Mapper接口 + XML） | 数据库CRUD |
| 数据库 | MySQL 8.0 | 数据存储 |
| 构建 | Maven | 依赖管理、打包成WAR |

### 2.2 分层架构图（画在纸上或口述）

```
浏览器 (JSP页面)
    ↓ HTTP请求
Controller层 (AuthController, AdminController, StudentController, HomeController)
    ↓ 调用
Service层 (UserService, ActivityService, SignupService, CheckinService, NoticeService)
    ↓ 调用
Mapper层 (UserMapper, ActivityMapper, SignupMapper, CheckinMapper, NoticeMapper)
    ↓ SQL
MySQL数据库 (user, activity, signup, checkin, notice 五张表)
```

### 2.3 为什么这样分层？（经典追问）

"这是经典的 MVC 分层架构。分层的好处是**职责分离、解耦合**：
- Controller 只管接收请求和跳转页面，不写业务逻辑
- Service 只管业务规则和事务控制
- Mapper 只管执行 SQL
- 这样如果我要换数据库，只改 Mapper 层；换前端，只改 JSP 和 Controller"

## 三、Spring MVC 请求流程（必问！）

### 3.1 标准答案

"一个请求的完整流程是：
1. 浏览器发 HTTP 请求到 `DispatcherServlet`
2. `DispatcherServlet` 根据 `@RequestMapping` 注解找到对应的 Controller 方法
3. Controller 方法调用 Service 处理业务逻辑
4. Service 调用 Mapper 查数据库
5. Controller 把结果放入 Model，返回视图名（比如 `student/activities`）
6. `InternalResourceViewResolver` 把视图名拼成 `/student/activities.jsp`
7. JSP 渲染页面返回给浏览器"

### 3.2 举个例子

"比如学生点击'查看活动列表'，浏览器请求 `/student/activities`，
DispatcherServlet 找到 `StudentController` 的 `@GetMapping("/activities")` 方法，
方法里调用 `activityService.listAllActivities()` 查出所有活动，
放到 Model 里返回视图名 `student/activities`，
最终渲染 `activities.jsp` 显示活动列表。"

### 3.3 配置在哪？

- **web.xml**（`WebContent/WEB-INF/web.xml`）：配置 `DispatcherServlet`，指定加载 `spring-mvc.xml` 和 `applicationContext.xml`
- **spring-mvc.xml**（`src/spring/`）：配置视图解析器（prefix=`/`、suffix=`.jsp`）、拦截器、文件上传解析器
- **applicationContext.xml**（`src/spring/`）：配置数据源、MyBatis SqlSessionFactory、Mapper扫描、事务管理器

## 四、数据库设计（必问！）

### 4.1 五张表

| 表名 | 作用 | 关键字段 |
|---|---|---|
| `user` | 用户表 | id, username(唯一), password, real_name, student_no, phone, role(student/admin/superadmin) |
| `activity` | 活动表 | id, title, content, location, start_time, end_time, deadline, max_count, current_count, status, cover_image, creator_id(外键→user) |
| `signup` | 报名表 | id, user_id(外键), activity_id(外键), signup_time, status(已报名/已取消/已签到/缺席)，**唯一约束(user_id, activity_id)** 防止重复报名 |
| `checkin` | 签到表 | id, signup_id(外键), user_id(外键), activity_id(外键), checkin_time, checkin_status, operator_id |
| `notice` | 公告表 | id, title, content, publisher_id(外键), create_time, status(有效/无效) |

### 4.2 表关系

```
user(1) ──< activity(N)     （一个管理员可以创建多个活动）
user(1) ──< signup(N)        （一个学生可以报名多个活动）
activity(1) ──< signup(N)    （一个活动可以被多人报名）
signup(1) ──< checkin(0或1)   （一条报名记录对应零或一条签到记录）
user(1) ──< notice(N)        （一个管理员可以发布多个公告）
```

### 4.3 如果问"为什么 signup 表要加唯一约束"

"为了在数据库层面防止同一个学生重复报名同一个活动。代码里也有查重逻辑（`getSignupByUserAndActivity`），但数据库约束是最后一道防线，保证数据完整性。"

### 4.4 如果问"为什么用外键"

"保证参照完整性。比如删除活动时，如果不先删掉关联的报名和签到记录，就会产生孤儿数据。我在 `ActivityService.deleteActivity` 里做了级联删除，先删 checkin → 再删 signup → 最后删 activity。"

## 五、核心功能在哪实现（必问！）

### 5.1 用户注册

| 环节 | 文件 | 关键方法 |
|---|---|---|
| 前端 | `WebContent/register.jsp` | 表单提交到 `/register` |
| Controller | `AuthController.java` | `register()` 方法，`@PostMapping("/register")` |
| Service | `UserService.java` | `register()` — 校验非空→查重→设默认角色student→保存 |
| Mapper | `UserMapper.java` + `UserMapper.xml` | `addUser` — INSERT 语句 |

**话术**："注册功能从 `register.jsp` 表单提交，到 `AuthController` 的 `register` 方法接收参数，构造 User 对象后调用 `UserService.register()`。Service 层先做参数校验（用户名密码不能为空），再查重（用户名是否已存在），最后调用 `UserMapper.addUser()` 写入数据库。"

### 5.2 用户登录

| 环节 | 文件 | 关键方法 |
|---|---|---|
| 前端 | `WebContent/login.jsp` | 表单提交到 `/login` |
| Controller | `AuthController.java` | `login()` — 调用 `userService.login()`，成功后存入 Session，按角色跳转 |
| Service | `UserService.java` | `login()` — 按 username 查用户，比对密码 |

**话术**："登录在 `AuthController.login()` 方法。先调用 `UserService.login(username, password)`，Service 里通过 `UserMapper.getUserByUsername` 查到用户后比对密码。如果验证通过，把 User 对象存到 HttpSession 里，然后根据角色跳转——管理员跳到 `/admin/index.jsp`，学生跳到 `/student/index.jsp`。"

### 5.3 活动管理（增删改查）

| 操作 | Controller方法 | Service方法 | Mapper方法 |
|---|---|---|---|
| 查列表 | `AdminController` `@GetMapping("/activities")` | `activityService.listAllActivities()` | `getAllActivities` |
| 新增/编辑 | `@PostMapping("/activity/save")` | `addActivity()` / `updateActivity()` | `addActivity` / `updateActivity` |
| 删除 | `@PostMapping("/activity/delete")` | `deleteActivity()` (**@Transactional**) | 级联删除 checkin→signup→activity |
| 查单个 | `@GetMapping("/activity/form")` | `getActivityById()` | `getActivityById` |

**删除的话术**："活动删除是带事务的（`@Transactional`）。因为一个活动关联了多条报名和签到记录，删除时要先删 checkin 表中该活动的签到记录，再删 signup 表的报名记录，最后才能删 activity 表本身。如果中间出错，事务会回滚，保证不会删一半。"

### 5.4 报名功能（重点！）

| 环节 | 文件 | 关键方法 |
|---|---|---|
| 前端 | `student/activities.jsp` | 点击"报名"按钮 |
| Controller | `StudentController.java` | `@PostMapping("/signup")` |
| Service | `SignupService.java` | `signupActivity()` — **@Transactional** |
| Mapper | `SignupMapper.java` + `SignupMapper.xml` | `getSignupByUserAndActivity`(查重) → `addSignup`(插入) → `ActivityMapper.increaseCurrentCount`(人数+1) |

**报名逻辑话术**："报名功能在 `SignupService.signupActivity()` 方法，加了 `@Transactional` 注解。逻辑是：
1. 先查 `getSignupByUserAndActivity` 看这个学生是否已经报过这个活动（查重）
2. 如果没报过，再查活动是否已满（`isActivityFull` — 比较 max_count 和 current_count）
3. 没满就 `addSignup` 插入一条报名记录，status 设为'已报名'
4. 最后调用 `increaseCurrentCount` 把活动的当前人数 +1
这四步是一个原子操作，任何一步失败都会回滚。"

### 5.5 签到功能

| 环节 | 文件 | 关键方法 |
|---|---|---|
| 前端 | `admin/checkin.jsp` | 管理员点"签到"按钮 |
| Controller | `AdminController.java` | `@PostMapping("/checkin")` |
| Service | `CheckinService.java` | `addCheckin()` |
| Mapper | `CheckinMapper.java` + `CheckinMapper.xml` | `getCheckinBySignupId`(查重) → `addCheckin`(插入) |

**签到话术**："签到在 `AdminController` 的 checkin 方法。管理员在签到页面看到某个活动下所有报名记录，点'签到'时传入 signupId。先调 `getCheckinBySignupId` 查这条报名是否已经签过到（防止重复签到），如果没有就创建 Checkin 记录，status 设为'已签到'，同时记录 operatorId（操作人即当前登录的管理员）。"

### 5.6 权限控制

**有两套权限控制（面试常问为什么）：**

1. **LoginInterceptor**（`interceptor/LoginInterceptor.java`）：Spring MVC 拦截器，在 `spring-mvc.xml` 中配置拦截 `/student/**` 和 `/admin/**`。preHandle 方法检查 Session 中是否有 currentUser，没有就跳转登录页；同时检查角色，学生不能访问 admin 路径，管理员不能访问 student 路径。

2. **AuthFilter**（`filter/AuthFilter.java`）：Servlet 原生 Filter，`@WebFilter` 注解拦截 `/admin/*` 和 `/student/*`。逻辑和 Interceptor 类似。

**话术**："权限控制用的是 Spring MVC 的 HandlerInterceptor。在 `spring-mvc.xml` 里配置了拦截 `/student/**` 和 `/admin/**` 路径。`LoginInterceptor.preHandle()` 方法先检查 Session 中有没有登录用户，没有就重定向到登录页。然后做角色校验——如果 URL 包含 `/admin/` 但用户角色不是 admin 或 superadmin，就重定向到学生页面，反之亦然。"

### 5.7 文件上传（活动封面图片）

**话术**："活动封面图片上传在 `AdminController` 的 `saveActivity` 方法里。用的是 Spring MVC 的 `MultipartFile` 接收文件。逻辑是：
1. 检查文件类型（只允许 jpg/jpeg/png/gif/bmp）
2. 检查文件大小（不超过 5MB，web.xml 里配了 multipart-config）
3. 用 UUID 生成新文件名（防止重名覆盖）
4. 保存到服务器的 `/uploads` 目录
5. 把文件路径存到 activity 表的 cover_image 字段"

### 5.8 CSV导出报名名单

**话术**："报名导出在 `AdminController` 的 `exportSignups` 方法。设置了 `Content-Type: text/csv` 和 `Content-Disposition: attachment` 让浏览器下载文件。手动拼 CSV 内容（报名ID、学号、姓名、手机号、报名时间、状态），加了 UTF-8 BOM 头防止中文乱码。"

## 六、MyBatis 关联映射（必问！这是亮点）

### 6.1 什么是关联映射

"MyBatis 的关联映射（`<association>`）可以在一次 SQL 查询中通过 JOIN 把多张表的数据查出来，自动映射到 Java 对象的嵌套属性上。比如查报名记录时，一次 JOIN 就能把关联的用户信息和活动信息都查出来。"

### 6.2 具体例子（SignupMapper.xml）

**SQL：**
```sql
SELECT s.*, u.username u_username, u.real_name u_real_name, ...,
       a.title a_title, a.location a_location, ...
FROM signup s
LEFT JOIN user u ON s.user_id = u.id
LEFT JOIN activity a ON s.activity_id = a.id
WHERE s.id = #{id}
```

**resultMap：**
```xml
<resultMap id="signupWithAllResultMap" type="Signup">
    <id property="id" column="s_id"/>
    <result property="userId" column="s_user_id"/>
    <!-- ... 其他 signup 字段 ... -->
    <association property="user" javaType="User">
        <id property="id" column="u_id"/>
        <result property="username" column="u_username"/>
        <!-- ... 其他 user 字段 ... -->
    </association>
    <association property="activity" javaType="Activity">
        <id property="id" column="a_id"/>
        <result property="title" column="a_title"/>
        <!-- ... 其他 activity 字段 ... -->
    </association>
</resultMap>
```

**话术**："以 `getSignupById` 为例。Signup 实体类里有 `private User user` 和 `private Activity activity` 两个关联对象。在 SignupMapper.xml 里定义了 `signupWithAllResultMap`，用 `<association>` 标签把 LEFT JOIN 查出的 user 和 activity 表的字段映射到这两个嵌套对象上。列别名用 `s_`、`u_`、`a_` 前缀区分三张表的同名字段。这样一次查询就能拿到完整的报名+用户+活动信息，不需要在 Java 代码里写三次查询。"

### 6.3 如果问"不用关联映射会怎样"

"传统 JDBC 方式需要先查 signup 表拿到 user_id 和 activity_id，再分别查 user 表和 activity 表，一共三次查询。用 MyBatis 关联映射只需要一次 JOIN 查询，减少了数据库访问次数，性能更好，代码也更简洁。"

### 6.4 如果问"LEFT JOIN 和 INNER JOIN 区别"

"用的 LEFT JOIN，因为即使 user 或 activity 记录被删除（理论上外键约束不会允许，但 LEFT JOIN 更安全），signup 记录本身仍然能查出来，关联对象会是 null。INNER JOIN 的话如果关联记录不存在，整条 signup 都查不出来。"

## 七、Spring 配置详解（如果问配置）

### 7.1 web.xml 做了什么

```
1. ContextLoaderListener — 启动时加载 applicationContext.xml（配置数据源、MyBatis、事务）
2. DispatcherServlet — Spring MVC 前端控制器，加载 spring-mvc.xml
3. CharacterEncodingFilter — 强制 UTF-8 编码，解决中文乱码
4. multipart-config — 配置文件上传大小限制（单文件5MB，总请求10MB）
5. session-config — 会话超时30分钟
```

### 7.2 applicationContext.xml 做了什么

```
1. 加载 db.properties 数据库配置
2. 扫描 com.clubsystem.service 包（@Service 注解的类）
3. 配置 DataSource（DriverManagerDataSource）
4. 配置 SqlSessionFactoryBean（整合 MyBatis + Spring）
5. MapperScannerConfigurer — 自动扫描 mapper 接口并生成代理实现
6. DataSourceTransactionManager + <tx:annotation-driven/> — 开启注解式事务
```

### 7.3 spring-mvc.xml 做了什么

```
1. 扫描 com.clubsystem.controller 和 interceptor 包
2. <mvc:annotation-driven/> — 开启 Spring MVC 注解支持
3. InternalResourceViewResolver — 视图解析器，prefix=/ suffix=.jsp
4. LoginInterceptor — 拦截 /student/** 和 /admin/** 做登录检查
5. StandardServletMultipartResolver — 文件上传支持
6. <mvc:default-servlet-handler/> — 让静态资源（CSS/JS/图片）由容器默认处理
```

## 八、事务管理（如果问 @Transactional）

### 8.1 哪里用了事务

| 方法 | 为什么需要事务 |
|---|---|
| `SignupService.signupActivity()` | 查重→检查容量→插入报名→更新人数，四步必须原子完成 |
| `ActivityService.deleteActivity()` | 删签到→删报名→删活动，三步级联删除必须原子完成 |

### 8.2 话术

"事务管理用的是 Spring 的 `@Transactional` 注解，在 `applicationContext.xml` 里通过 `<tx:annotation-driven/>` 开启。事务管理器是 `DataSourceTransactionManager`。以报名为例，`signupActivity` 方法加了 `@Transactional`，里面做了四步操作：查重、检查容量、插入报名记录、更新活动人数。如果第四步失败了，前三步也会回滚，不会出现'报名记录插了但人数没加'的不一致状态。"

## 十、可能被问到的刁钻问题

### Q: 为什么 dao 包和 mapper 包同时存在？

"dao 包是作业1阶段用纯 JDBC（DBUtil + PreparedStatement）写的，作业2 引入了 MyBatis 后新增了 mapper 包。两个包功能有重叠，但作业2 的重点在 MyBatis，实际业务调用走的是 Service → Mapper 这条链路。dao 包保留了但不再被 Service 调用。"

### Q: 你的密码是明文存储的，不安全吧？

"是的，当前版本密码是明文比对（`UserService.login` 里直接比较）。如果要改进，应该用 BCrypt 或 MD5+盐值加密存储密码。这是课程设计的简化处理，实际项目中不会这样。"

### Q: Spring MVC 和 Servlet 是什么关系？

"Spring MVC 的核心是 `DispatcherServlet`，它本身就是一个 Servlet，在 web.xml 里配置。它做的事情是：把请求分发到各个 Controller 方法（通过 @RequestMapping 匹配 URL），代替了传统 Servlet 模式中每个功能写一个 Servlet 的做法。所以 Spring MVC 是对 Servlet 的封装和增强。"

### Q: MyBatis 和 JDBC 是什么关系？

"MyBatis 底层就是基于 JDBC 的。它做的事情是：把 SQL 写在 XML 里，自动完成 ResultSet 到 Java 对象的映射（resultMap），自动处理参数绑定（#{} 占位符），自动管理 SqlSession 生命周期。省去了 JDBC 中手写 PreparedStatement、手动 set 参数、手动 rs.getString() 的重复代码。"

### Q: 为什么用 LEFT JOIN 不用 INNER JOIN？

"LEFT JOIN 保证即使关联的 user 或 activity 记录不存在，signup 记录本身仍然能查出来，只是关联对象为 null。更安全。INNER JOIN 如果关联记录不存在，整行都会被过滤掉。"

### Q: current_count 字段会不会和 signup 表的实际数量不一致？

"理论上在事务保护下不会。`signupActivity` 方法里插入 signup 记录和 increaseCurrentCount 是在同一个事务里的。但如果有人直接在数据库里手动改数据，可能会不一致。更严谨的做法是每次查报名人数时用 `SELECT COUNT(*) FROM signup WHERE activity_id=?` 来实时计算，但性能较差。用 current_count 字段是空间换时间的做法。"

### Q: 你们的拦截器和过滤器有什么区别？

"Filter 是 Servlet 规范的，在请求到达 Servlet 之前拦截；Interceptor 是 Spring MVC 的，在请求到达 Controller 方法之前拦截。我这个项目里两个都配了，功能有重叠。如果只用一个的话，用 Spring 的 Interceptor 更合适，因为它能拿到 HandlerMethod，和 Spring 生态整合更好。"

### Q: 为什么不用 Spring Boot？

"课程要求学习 Spring MVC 的底层配置，包括 web.xml、applicationContext.xml、spring-mvc.xml 这些手动配置。Spring Boot 帮你自动配置了这些，但课程目的是理解原理，所以用手动配置的方式。理解了这些配置，用 Spring Boot 时就知道自动配置在做什么。"

1. "我的项目是校园社团活动报名与签到管理系统"
